#01
hasValidCoordinates(stop) called with wrong argument type — coordinates display always broken
FieldVisitPlanner.jsx · Stop card render · ~line 2730
Critical
The function hasValidCoordinates(lat, lng) expects two numeric arguments, but the stop card renders pass the entire stop object as the first argument (hasValidCoordinates(stop)) in three places: the Tooltip title, the cursor style, and the text color. Number(stop) evaluates to NaN, so Number.isFinite(NaN) returns false and the function always returns false for all three calls.
Impact: Even when a stop has perfectly valid saved coordinates, the entity name text is never clickable/blue, the tooltip always shows "No coordinates yet" instead of "Click to focus on map", and the cursor never changes to pointer. Visually breaks the coordinate-focus interaction for every stop that has an entity name. 
// ❌ Wrong — stop object passed instead of lat/lng
title={hasValidCoordinates(stop) ? 'Click to focus on map' : 'No coordinates yet'}
cursor: hasValidCoordinates(stop) ? 'pointer' : 'default'
color: hasValidCoordinates(stop) ? '#1677ff' : undefined

// ✅ Fix — pass the actual coordinates
title={hasValidCoordinates(stop.latitude, stop.longitude) ? 'Click to focus on map' : 'No coordinates yet'}
cursor: hasValidCoordinates(stop.latitude, stop.longitude) ? 'pointer' : 'default'
color: hasValidCoordinates(stop.latitude, stop.longitude) ? '#1677ff' : undefined

#02
Side effect (setTimeout) called inside a setStops state updater — React anti-pattern
FieldVisitPlanner.jsx · addDraftStops() · ~line 1770
Important
Inside the addDraftStops function, setTimeout(() => geocodeQueue.forEach(q => autoGeocodeStop(...)), 0) is called from within the functional updater passed to setStops(prev => { ... }). React may call state updater functions more than once (in StrictMode, twice), and updater functions are supposed to be pure. A side-effect inside an updater can lead to double-geocoding requests in development (Strict Mode double-invoke) and unpredictable behavior in production.
Impact: In React 18 Strict Mode, the updater runs twice, potentially firing two geocoding requests per stop. Even outside Strict Mode, having a side effect inside a state updater is a React rule violation that can cause subtle races if React batches or defers the update. 
// ❌ Side effect inside state updater
setStops(prev => {
  ...
  if (geocodeQueue.length > 0) {
    setTimeout(() => geocodeQueue.forEach(q => autoGeocodeStop(q.idx, q.hints, q.countryCode)), 0);
  }
  return newStops;
});

// ✅ Fix — hoist the setTimeout outside the setStops call
setStops(prev => {
  ...
  return newStops;
});
if (geocodeQueue.length > 0) {
  setTimeout(() => geocodeQueue.forEach(q => autoGeocodeStop(q.idx, q.hints, q.countryCode)), 0);
}

Important

#04

Important
When the primary match (by country names in from_label/to_label) fails, the fallback returns the first leg that has any dep_datetime and arr_datetime — regardless of origin or destination. For a trip with multiple legs (e.g. DXB→RUH and RUH→CAI), the fallback will always return Leg 1 for every cross-country pair that doesn't match by country name.

  return (from.includes(fromCountry) || fromCountry.includes(from.split(' ')[0])) &&

#05
markArrived blocks check-in for overdue stops (past scheduled date)
FieldVisitInTrip.jsx · markArrived() · ~line 175
Important
The check-in guard compares the stop's visit_date against today's date and returns early if they don't match — including past dates. A sales rep who postponed a stop yesterday due to poor signal or a schedule delay can never GPS-check-in to that stop, even though the trip is still active and the stop still needs to be marked.
Impact: Legitimate late check-ins (travel delays, rescheduled meetings) are permanently blocked by a hard date equality check. The rep is forced to use manual outcome logging with no GPS record, losing location audit data. 
// ❌ Hard equality: only allows check-in on exact scheduled day
if (stop.visit_date) {
  const today = dayjs().startOf('day').format('YYYY-MM-DD');
  const stopDate = dayjs(stop.visit_date).format('YYYY-MM-DD');
  if (stopDate !== today) {
    message.warning(`This stop is scheduled for ${stopDate}...`);
    return;
  }
}

// ✅ Fix — allow check-in if stop is overdue (past) or today
if (stop.visit_date) {
  const stopDate = dayjs(stop.visit_date).startOf('day');
  const today = dayjs().startOf('day');
  if (stopDate.isAfter(today)) {
    message.warning(`This stop is scheduled for a future date (${stopDate.format('DD MMM')}). Check-in is only allowed on or after the visit date.`);
    return;
  }
}

#06
Steps navigation allows jumping to Step 2 without ever adding stops
FieldVisitPlanner.jsx · <Steps onChange> · ~line 2470
Important
The Steps component's onChange calls canAdvance() to gate forward navigation. But canAdvance() only checks Step 0 conditions when step === 0. If the form title/dates are filled and the user clicks the "Review & Save" step header directly from Step 0, canAdvance() returns true and they jump to Step 2 with zero stops. The "Save Trip" button is disabled by the GPS-unresolved count, but the user sees no message explaining why — they just see a disabled button.
Impact: Confusing UX: user can reach the Review page with an empty stop list, no clear explanation, and a disabled Save button. The "Next" button inline validates stops correctly, but the Steps header bypasses it. 
// ❌ canAdvance() only checks current step — jumping 0→2 checks step 0 only
onChange={(s) => { if (s < step || canAdvance()) setStep(s); }}

// ✅ Fix — validate all intermediate steps when jumping forward
onChange={(s) => {
  if (s <= step) { setStep(s); return; }  // always allow going back
  // Validate every step between current and target
  const savedStep = step;
  for (let check = savedStep; check < s; check++) {
    const tmpStep = step; // canAdvance reads `step` state
    if (check === 0 && !canAdvance()) return;  // form validation
    if (check === 1 && stops.length === 0) { message.warning('Add at least one stop.'); return; }
  }
  setStep(s);
}}

FieldVisitReviewStep.jsx · filtered stops .map() · ~line 130
Medium
When reviewShowUnsavedOnly is on, the component filters stops and then calls stops.indexOf(s) to get the original index. This relies on object reference identity. Any state update that recreates stop objects (a shallow spread like { ...stop, someField }) would cause indexOf to return -1, displaying actualIndex + 1 = 0 for all filtered stops (showing "Stop 0") and making the ETA chain look up etaChain[-1] (undefined). 
// ❌ Reference equality is fragile after any state update that recreates objects
const actualIndex = stops.indexOf(s);

// ✅ Fix — match by a stable identity field
const actualIndex = stops.findIndex(orig =>
  orig.local_id ? orig.local_id === s.local_id : orig.id === s.id
);
